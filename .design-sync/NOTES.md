# design-sync notes

- **Known render warn — `CountUpCard` screenshots may show a slightly-under number.** The component animates its displayed value from 0 up to the target over 0.8s (Framer Motion `animate()`, driven by `requestAnimationFrame`). `package-capture.mjs`'s `settle()` only awaits `document.fonts.ready` + image decode — it does not wait for CSS/JS animations to finish, and `page.clock.setFixedTime()` pins `Date.now()` but does not freeze `requestAnimationFrame`. So a captured screenshot typically lands ~80-90% through the count-up (e.g. `value={18}` may show as `14.9`). This is a capture-timing artifact, not a broken component or preview — the real app always settles correctly once mounted for more than ~1s. Not fixable from a preview `.tsx` (no prop to disable the animation) or via config override (`package-capture.mjs` is a top-level orchestration script, not a forkable `lib/*.mjs` adapter). Accepted as a known limitation rather than chased further; re-syncs should not treat this as a new regression if seen again on `CountUpCard`'s cells.
- **Install/build must run from `/root/build`, not the canonical mount** (`/mnt/Gemini Cli/build`). `npm ci`/`npm install` on the mount fails with `EACCES` on `symlink` — the Android storage mount doesn't support symlinks, which npm needs for `node_modules/.bin`. This mirrors the existing dual-folder dev workflow (see `scripts/dev-lan.sh` / project memory): edit source on the mount, sync (`tar` copy, excluding `node_modules`/`.git`/`dist`) to `/root/build` (ext4), and run all node/npm/vite tooling there.
- Package manager: npm (`package-lock.json` present). No `engines.node` pin in `package.json`; system node is v22.23.1.
- No Storybook anywhere in this repo (confirmed via `.storybook`/`*.stories.*` search) — user confirmed none exists elsewhere either. Shape = `package`.
- **`package.json` is `"private": true` with no `main`/`module`/`exports`** — this app has no library build (`vite build` produces the whole-app bundle, not an importable component entry). Synth-entry mode is used (converter reads `src/components` directly, no `dist/`). `.d.ts` contracts will be weaker than a real build would give — this is inherent to the repo, not a config gap.
- **Only 7 real reusable components exist**: `NavBar`, `Sidebar`, `AlertsPanel`, `CountUpCard`, `Typeahead`, `components/erp/ModuleWorkspace`, `components/erp/ImportWizard`. There is NO Button/Card/Badge/Dialog/Table primitive layer — the app's "consistent cards/forms/badges" (per its `CLAUDE.md`) come from repeating the same Tailwind utility classes directly in page files, not shared components. User confirmed (2026-07-09) to sync exactly these 7 as-is rather than inventing new primitive components — that would violate this skill's "ship what the customer already built, never a reimplementation" principle. **Re-sync risk**: if the app later grows a real component layer (Button/Card/etc. get extracted), re-run discovery — don't assume this 7-component list is still complete.
- Preview scope: user chose to author previews for all 7 (small set, full authoring is cheap here) rather than floor-carding any of them.
- **Fonts**: `Outfit` is loaded via a Google Fonts `<link>` in `index.html` (runtime CDN load, not a shipped local file) — `tailwind.css`'s `font-family: 'Outfit', 'Inter', sans-serif` references it but the converter's CSS scrape can't see an HTML `<link>`. `Inter` is a fallback name in that same stack that's never actually loaded anywhere in the repo (no link, no `@font-face`). User confirmed (2026-07-09) to mark both via `cfg.runtimeFontPrefixes` rather than shipping a local Outfit woff2 — accept substitute/system-font rendering in the DS pane rather than adding a local font file. **Re-sync risk**: if the app later self-hosts Outfit (e.g. moves off the Google Fonts CDN), this should switch to `cfg.extraFonts` pointing at the shipped woff2 instead.
- **Unrelated app bug discovered incidentally**: `CountUpCard.tsx` references CSS classes `concrete-accent-orange`/`glow-text-orange` (default) and `concrete-accent-cyan`/`glow-text-cyan` — none of which exist anywhere in `src/styles/tailwind.css` (only `-gray` and `-red` variants are defined). The "glow" accent effect silently no-ops for every CountUpCard except the `accent="red"` case. This is a real pre-existing bug in the app, unrelated to design-sync — flagged here rather than fixed, since fixing app source is outside this sync's scope. The conventions.md file only documents the classes that actually exist (`-gray`/`-red`), not the broken references.
- **Bundle-export bug found and forked around**: see `.design-sync/overrides/source-kit.mjs` (`libOverrides` in config.json has the full reason) — this repo's components are all `export default`, and the stock synth-entry writer's `export * from` never re-exports `default` bindings, so the bundle shipped 0 working components until this fork. **Re-sync risk**: if design-sync's bundled `lib/source-kit.mjs` is updated upstream, diff it against this fork (per the skill's re-sync instructions) and merge forward — don't let the fork silently drift from upstream fixes.
- **4 of 7 components ship as floor cards only (no authored preview), by design**: `NavBar`, `Sidebar`, `components/erp/ModuleWorkspace`, `components/erp/ImportWizard` all call `useAuth()`, which reads `AuthContext` (`src/lib/useAuth.tsx`) and **throws** if not wrapped in the real `<AuthProvider>`. That provider's state comes from a live `supabase.auth.getSession()` call with no way to inject a fake session via props. Faking it would require either shipping real production Supabase credentials in the design-system bundle (credential-leak risk, refused) or modifying the app's real `useAuth.tsx` to export new plumbing purely for this sync (a production source change, needs explicit user sign-off, not taken here). User confirmed (2026-07-09) to floor-card these 4 rather than pursue either option. **Re-sync risk**: if the app later adds a test/demo auth mode of its own (unrelated to this sync), these 4 could get real authored previews — check back rather than assuming this is permanent.
- Authored previews exist for the other 3: `AlertsPanel` (pure `alerts: Alert[]` prop), `CountUpCard` (pure numeric props), `Typeahead` (its Supabase autocomplete query only fires once its `value` prop is non-empty, so an idle-state story never touches the network — the "suggestions open" state is not authored since it needs a live query result and can't render statically; noted here per the skill's guidance for un-renderable states).
- **Required setup step, not committed**: `package-build.mjs` resolves `PKG_DIR` via `join(NODE_MODULES, PKG)` when no `--entry` is passed (needed for synth-entry mode — see above), but this repo isn't installed as its own dependency, so `node_modules/safetech-control-center` doesn't exist by default and the build crashes with `ENOENT: .../node_modules/safetech-control-center/package.json`. Fix: `ln -sfn .. node_modules/safetech-control-center` (self-referencing symlink, run once from `/root/build`) before the first `package-build.mjs`/`resync.mjs` call. This is a `node_modules` entry so it's gitignored and must be recreated on every fresh clone/toolchain-copy sync — same category as the `.design-sync/node_modules` fork symlink already required for `.design-sync/overrides/source-kit.mjs`'s bare `ts-morph` import.
- **CRITICAL — `cssEntry` must point at a COMPILED stylesheet, not the raw source.** `src/styles/tailwind.css` on disk is unprocessed Tailwind source (`@tailwind base; @tailwind components; @tailwind utilities;`) — Vite/PostCSS expands it at the app's own build time. The design-sync converter does NOT run Tailwind/PostCSS itself; it just copies whatever `cssEntry` points at verbatim. Pointing `cssEntry` at the raw source (the auto-detected default) shipped **zero actual utility-class CSS** — every preview rendered as completely unstyled plain text (confirmed via first-pass grading screenshots: no colors, no borders, no padding, no dark theme — just bare text on white). Fixed by adding `cfg.buildCmd` (`npx tailwindcss -i src/styles/tailwind.css -o .design-sync/.cache/tailwind-compiled.css --config tailwind.config.cjs --minify`) and pointing `cssEntry` at that compiled output instead. **Re-sync must always re-run `cfg.buildCmd` before `package-build.mjs`** — Tailwind's JIT scan (`content: ['./index.html', './src/**/*.{ts,tsx,js,jsx}']` in `tailwind.config.cjs`) only includes classes actually used in the source at compile time, so a stale compiled CSS silently drops classes added by newer components/pages. This is the single most important re-sync risk in this whole sync — an agent who skips `buildCmd` and just re-runs the converter will silently ship unstyled previews again.

## Re-sync risks (checklist for the next sync)

1. **Always re-run `cfg.buildCmd`** (compiles Tailwind CSS) before the converter — see the CRITICAL note above. Skipping this silently ships unstyled previews.
2. **Always sync mount → `/root/build` before running any node/npm/tooling** — the mount can't run node (`EACCES` on symlink). This project's `tar -cf ... -C <mount> . | tar -xf ... -C /root/build` pattern (or equivalent) must run first, every time.
3. **Recreate two gitignored symlinks on a fresh clone/toolchain-copy** (both are `node_modules` entries, never committed): `ln -sfn .. node_modules/safetech-control-center` (PKG_DIR resolution) and `ln -sfn ../.ds-sync/node_modules .design-sync/node_modules` (fork's `ts-morph` import).
4. **Re-run component discovery** if the app's `src/components/` grows a real primitive layer (Button/Card/etc.) — the current 7-component scope and the "no primitive layer" framing in `conventions.md` would both go stale.
5. **Diff `.design-sync/overrides/source-kit.mjs` against the upstream `lib/source-kit.mjs`** if design-sync itself gets updated — the fork could silently miss an upstream fix.
6. **Re-check the floor-carded 4** (`NavBar`, `Sidebar`, `ModuleWorkspace`, `ImportWizard`) — only revisit if the app adds some kind of test/demo auth mode; don't assume the auth blocker lifted on its own.
7. **`conventions.md` is user-owned** — re-syncs should validate its claims against the fresh build (per the base skill's rule) and report drift, never silently rewrite it.
